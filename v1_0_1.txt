10.07 Ревизия целей v1.0 по факту текущего состояния кода графа 


	[РЕШЕНО] Указание инструментов из registry в prompts (главная цель прошлой версии)
		Реализовано: PLANNER_SYSTEM_PROMPT получает {corpus_registry}, EXECUTION_PLANNER_SYSTEM_PROMPT и EVIDENCE_AGGREGATOR_SYSTEM_PROMPT получают {registry_description} через get_registry_description() из tools/registry.py.
		Галлюцинации несуществующих инструментов у evidence_aggregator, которые раньше уводили граф в бесконечный цикл, в текущей версии не наблюдаются.


	1. Тестирование и деплой первой версии

		1) [РЕШЕНО] Улучшение промптов (инструменты указаны явно) - см. пункт выше.

		2) [ЧАСТИЧНО] Логи для тестов + визуализация в консоли
			Сделано: подробное print()-логирование добавлено во все узлы (Router, Planner, ExecPlanner, Orchestrator, Aggregator) + loguru-логгер в assistant_node.
			Не доделано: код сохранения graph_visualization.png (draw_mermaid_png) в graph.py закомментирован - визуализация есть только как ручной артефакт, не встроена в пайплайн.

		3) [ЧАСТИЧНО] Тестирование на сложных запросах
			Сделано: graph.py переведён с одноразового invoke на интерактивный CLI-цикл (while True + input()) - появилась инфраструктура для ручных прогонов разных вопросов в одной сессии.
			Не доделано: самих зафиксированных тестовых кейсов/результатов на сложных запросах пока нет.

		4) [НЕ СДЕЛАНО] Деплой на сервере - агент по-прежнему CLI only

		5) [РЕШЕНО] Загрузка на гитхаб

		6) [НЕ СДЕЛАНО] Презентация


	2. [АКТУАЛЬНО, БЕЗ ИЗМЕНЕНИЙ] Узел уточнения (Human-in-the-Loop)
		mode в state/schema.py по-прежнему бинарный: "research" / "chat". Состояния ask_user или механизма interrupt в графе нет.
		Planner всё так же обязан сам додумывать гипотезы и сужать абстрактные цели пользователя ("найди что-нибудь интересное про слово 'крипта'") вместо того, чтобы остановиться и переспросить. Переносится в v1.0.1 без изменений.


	3. [ЧАСТИЧНО] Обработка пагинации и массивов данных
		Сделано: в ResearchState уже заведено поле pagination_context: dict - место под состояние пагинации зарезервировано в схеме.
		Не доделано: сама логика ("получено 10 примеров из 5000, нужен ещё вызов с page=2 / sampling=100") не реализована ни в execution_planner, ни в nkrja_client, ни в orchestrator. 


	4. [РЕШЕНО] Краткосрочная память сессии (Checkpointer)
		graph.py подключает MemorySaver, весь CLI-цикл работает через единый thread_id ("cli-session-1").
		Одноразовые поля (iteration_count, is_goal_reached, needs_replanning, planned_actions)  сбрасываются перед каждым новым вопросом, накопительные (facts, evidence, hypotheses, completed_actions) - нет.
		planner_node официально умеет определять mode="chat" на основе накопленных фактов из предыдущего цикла и отвечать на уточнения без похода в API. Пункт полностью закрыт.


	5. [АКТУАЛЬНО, БЕЗ ИЗМЕНЕНИЙ] Прозрачность методологии (Explainability)
		assistant_node по-прежнему получает только плоский список facts и пишет ответ одним блоком - отдельного раздела "Методология" (какие корпуса запрошены, какие фильтры/resultType применены, сколько документов проанализировано) в промпте и коде нет.
		Все нужные для этого данные уже лежат в массиве evidence (action, params, response) - реализация не начата, переносится в v1.0.1.


	Новые цели v1.0.1:

	6. Векторная БД и RAG
		Проблема: facts копятся в ResearchState как плоский список строк и целиком передаются в LLM на каждом шаге (evidence_aggregator, assistant_node).  это не масштабируется по токенам. Память при этом ограничена рамками одного thread_id и живёт только пока жив процесс (MemorySaver чисто in-memory).
