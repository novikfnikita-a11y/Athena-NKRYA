# Хендофф: две архитектурные доработки NKRJA Research Agent

Этот документ адресован модели/агенту, который уже обладает полным контекстом кодовой базы проекта (graph.py, planner.py, execution.py, orchestrator.py, evidence.py, assistant.py, schema.py, client.py, registry.py, nkrja_client.py). Ниже — не постановка задачи с нуля, а объяснение двух конкретных концепций доработки, которые уже были проработаны и согласованы с автором проекта. Задача читающей модели — понять концепции достаточно глубоко, чтобы реализовать их в коде, не переспрашивая базовые мотивы.

---

## 0. Диагноз текущей архитектуры (общий для обеих концепций)

Граф: `planner → {execution_planner | assistant}` (по mode) → `execution_planner → api_orchestrator → evidence_aggregator` (циклически, до 4 итераций или до `is_goal_reached`) → `assistant → END`.

Пять конкретных проблем в текущем коде, из которых растут обе концепции:

1. **`evidence.py`**: сырые ответы API складываются в `evidence` как есть, а усечение происходит только потом — по границе блоков, на потолке в 90 000 символов **одного батча**. При параллельном сравнении нескольких корпусов это узкое место — все корпуса конкурируют за один и тот же лимит.
2. **`schema.py`**: `facts: Annotated[list[str], operator.add]` — плоский, неограниченно растущий список **непрозрачных строк**. Нет привязки факта к тому, о каком корпусе/лемме он был. `assistant.py` берёт этот список целиком (`"\n".join(...)`) и скармливает LLM без ранжирования и без структуры — из-за этого модель физически не может корректно держать в голове, какой факт к какому корпусу относится, если сравнивается несколько корпусов.
3. **`client.py`**: `max_tokens=2048` — это потолок на **выходе** финального узла. Даже при идеальных входных данных содержательный сравнительный отчёт с таблицами по 3 корпусам может не поместиться.
4. Парсинг JSON-ответов LLM всюду (`planner.py`, `execution.py`, `evidence.py`) сделан через `response.content.strip().replace("```json"...)` + `re.search(r'\{.*}')` + `try/except` — хрупко, в проде это будет падать чаще, чем на MVP-нагрузке.
5. Нет механизма **fan-out** для независимых под-задач. Если пользователь просит сравнить лемму сразу по нескольким корпусам, вся система всё равно прогоняет один линейный цикл с одним общим `iteration_count` и одним общим массивом `evidence` — архитектурно это не рассчитано на параллельные независимые ветки исследования.

## Концепция 1: Fan-out + сжатие на входе + типизированные факты + reduce-синтез

### Цель концепции
Сделать физически возможным корректное сравнительное исследование по нескольким сущностям/корпусам одновременно (пример: лемма «мать» в корпусах MAIN, POETIC, SPOKEN), не теряя данные на усечении и не полагаясь на то, что LLM сама удержит в контексте, какой факт к какому корпусу относится.

### Компоненты и логика

**1.1. Планировщик как декомпозитор запроса, а не только классификатор режима.**
Сейчас `planner_node` определяет только `mode: research/chat`. Нужно расширить его так, чтобы он детектировал **множественность независимых сущностей** в запросе (несколько корпусов для одной леммы, несколько лемм, и т.п.) и вместо одного плоского `goal/hypotheses/research_plan` возвращал список под-целей:

```python
sub_goals = [
    {"lemma": "мать", "corpus": "MAIN",   "goal": "...", "hypotheses": [...], "research_plan": [...]},
    {"lemma": "мать", "corpus": "POETIC", "goal": "...", "hypotheses": [...], "research_plan": [...]},
    {"lemma": "мать", "corpus": "SPOKEN", "goal": "...", "hypotheses": [...], "research_plan": [...]},
]
```

**1.2. Fan-out выполнение через `Send` API LangGraph.**
Для каждой под-цели запускается независимый прогон тройки `execution_planner → api_orchestrator → evidence_aggregator`, каждый — со своим локальным срезом состояния: собственный `iteration_count`, собственный `evidence`, собственный `facts`, собственный лимит `max_iterations=4`. Это устраняет конкуренцию за общий 90k-лимит: у каждой ветки (по одному корпусу) — свой независимый бюджет на данные, вместо того чтобы 3 корпуса делили один и тот же батч-лимит.

**1.3. Сжатие на входе (map-step) — вместо усечения по границе блока.**
Ключевое изменение порядка операций: сейчас сырой JSON от `NKRJAClient` (полный конкорданс, полная статистика, полный скетч) складывается в `evidence` как есть, и только потом режется по размеру. Правильно — сжимать **каждый отдельный ответ API немедленно при получении**, ещё до попадания в общий контекст, отдельным узлом или шагом внутри оркестратора (`evidence_compressor`). Правила сжатия по типу `resultType`:
   - `PORTRAIT_CONCORDANCE` (список примеров-предложений) → top-k репрезентативных примеров + выявленные паттерны, а не полный список;
   - `PORTRAIT_STATS` (распределение по метаданным) → агрегированная сводка/тренд, а не сырая таблица;
   - `PORTRAIT_SKETCH` / `PORTRAIT_SIMILAR` → top-N коллокаций/похожих слов с весами, а не полный список.

   Это map-reduce над длинным контентом: map — сжать каждый источник по отдельности (дешёвый маленький вызов LLM или детерминированная логика), reduce — синтез уже из сжатых кусков. В результате объём данных на каждый вызов API становится O(summary), а не O(raw_response), и усечение по границе блока в 90к перестаёт быть узким местом в типичном случае.

**1.4. Факты — типизированные объекты, а не строки.**
Замена `facts: list[str]` на структуру:

```python
class Fact(TypedDict):
    lemma: str
    corpus: str
    metric: str        # "frequency", "collocation", "first_mention", ...
    value: Any
    source_action: str
    evidence_id: str    # ссылка на сырой (сжатый) артефакт для drill-down
```

Это позволяет **программно**, без участия LLM, собрать сравнительную таблицу (сгруппировать по `metric`, разложить по `corpus` в колонки) до того, как модель вообще начнёт писать текст. Убирает риск, что модель «потеряет» один из корпусов при синтезе, и попутно даёт бесплатную трассируемость/цитируемость фактов.

**1.5. Общее хранилище фактов (vector + relational) вместо дампа всего списка в промпт.**
Все `Fact` из всех веток пишутся в общее хранилище (для MVP достаточно SQLite/лёгкого векторного индекса; в README проекта уже есть roadmap на Chroma/Qdrant/pgvector — эту концепцию стоит вводить сейчас, а не откладывать). `assistant_node` и новый узел синтеза не берут «все facts», а делают **scoped retrieval**: для известного сравнительного запроса — тривиальный SQL-фильтр по `(lemma, corpus IN [...])`, без всякого semantic search; semantic search нужен только для более расплывчатых follow-up вопросов в режиме `chat` после накопления фактов за много сессий.

**1.6. Отдельный узел-синтезатор (reduce) перед `assistant_node`.**
После схождения всех веток отдельный узел берёт уже собранную программно таблицу фактов и формирует содержательное сравнение (где частотность выше, чем отличаются коллокации и т.д.) — это отдельный LLM-вызов, разделяющий задачу «сравнить цифры» и задачу «красиво написать текст». `assistant_node` дальше только оформляет уже готовый сравнительный анализ в финальный ответ, а не одновременно синтезирует и пишет прозу в одном вызове с текущим лимитом 2048 токенов на выходе.

**1.7. Разделение бюджетов токенов.**
Два конфига LLM: дешёвый/быстрый с маленьким `max_tokens` — для extraction/compression вызовов (их много, по одному на API-ответ, по всем веткам параллельно); отдельный с увеличенным `max_tokens` (4000–8000+) — для финального синтеза/ответа, поскольку полноценный сравнительный отчёт с таблицами не влезает в нынешние 2048.

**1.8. Точечные фиксы, не требующие полного фан-аут переписывания (можно внедрять уже сейчас):**
   - `client.py`: поднять `max_tokens` для финального LLM-инстанса;
   - `planner.py` / `execution.py` / `evidence.py`: заменить хрупкий regex-парсинг JSON на structured output / function calling, если прокси VseGPT/DeepSeek это поддерживает — критично для продовой надёжности;
   - `orchestrator.py`: добавить кэш по хэшу `(action, params)` — избежать повторных вызовов API НКРЯ и повторной LLM-компрессии для пересекающихся сравнительных запросов;
   - `graph.py`: заменить `MemorySaver` на `PostgresSaver`/`SqliteSaver` — сессии должны переживать рестарт процесса.

### Производный набор фич для боевого сайта НКРЯ
- drill-down из любого `Fact` к его сырому (сжатому) артефакту по `evidence_id`;
- экспорт сравнения в CSV/таблицу;
- human-in-the-loop уточнение (`interrupt` в LangGraph), когда планировщик не уверен, какие корпуса имелись в виду;
- observability/трейсинг (LangSmith или аналог) — обязателен при параллельных ветках, иначе отладка «какая ветка вернула мусор» становится нетривиальной;
- rate limiting / cost guard на количество веток и вызовов на ветку — fan-out умножает объём LLM- и API-вызовов на N;
- единый Pydantic-слой валидации параметров инструментов для всех tool'ов реестра (сейчас защита через `RESULTTYPE_CORPUS_WHITELIST` есть только для `get_word_portrait` в `orchestrator.py`).

## Концепция 2: Trace / «thinking-режим» для живой видимости мыслей и действий агента

### Цель концепции
Дать пользователю сайта живую видимость того, что агент делает и «думает» на каждом шаге исследовательского прогона — по духу UX похоже на extended thinking режим современных LLM-чатов, но адаптировано под систему с несколькими распределёнными узлами (несколько отдельных LLM-вызовов + реальные вызовы инструментов), а не единый непрерывный chain-of-thought одной модели.

### Важное разделение, которое нужно сохранить в реализации
1. **Пост-фактум структурированный reasoning** (уже существует в коде, сейчас просто выбрасывается): `planner.py` уже вычисляет `mode_reasoning`, `corpus_reasoning`; `execution.py` уже вычисляет `reasoning` для выбора инструмента; `evidence.py` уже вычисляет `aggregator_reasoning` для решения «цель достигнута / нужен реплан». Всё это сейчас просто печатается в консоль (`print(...)`) и никуда не сохраняется и не показывается пользователю. Это самый дешёвый и безрисковый первый шаг — не создавать новую генерацию reasoning, а просто перестать её выбрасывать.
2. **Настоящий token-by-token chain-of-thought** (опционально, дороже) — требует переключения конкретных узлов на reasoning-модель (у DeepSeek есть R1-класс, `deepseek-reasoner`, который возвращает отдельное поле `reasoning_content`, отдельно от `content` — аналог thinking-блоков) вместо текущей `deepseek-chat` в `client.py`. Рекомендация: включать это **точечно** только для `planner` и `evidence_aggregator` — узлов с настоящим качественным суждением («это одна исследовательская задача или две?», «цель достигнута, или методология в целом неверна?»). НЕ включать для `execution_planner` — это механический выбор инструмента по фиксированному реестру возможностей, где reasoning-модель добавляет стоимость/задержку без содержательного выигрыша в качестве решения.

### Архитектурное требование: трасса НЕ должна попадать в `ResearchState`
Критично: нельзя добавлять поле трассы/лога в `ResearchState` через `operator.add`, потому что состояние графа LangGraph — это именно то, что скармливается в промпт каждого узла. Добавление вечно растущего списка трассы в state воспроизведёт ту же самую проблему раздувания контекста, которая обсуждалась в Концепции 1, только теперь трасса начнёт попадать обратно в LLM-вызовы. Данные трассы должны быть **только для пользователя/фронтенда**, структурно отделены от рабочей памяти, которая управляет решениями графа и промптами.

### Конкретный дизайн

**2.1. Схема `TraceEvent`** (сайд-канальная модель данных, словарь в терминах ReAct: Thought / Action / Observation / Decision / FinalAnswer):

```python
class TraceEvent(TypedDict):
    run_id: str
    thread_id: str
    branch_id: str | None   # привязка к веткам fan-out из Концепции 1, для группировки в UI по корпусу
    iteration: int
    node: str                # "planner" | "execution_planner" | "api_orchestrator" | "evidence_aggregator" | "assistant"
    type: Literal["thought", "action", "observation", "decision", "final_answer"]
    content: str | dict
    ts: float
```

**2.2. `emit_trace()` — тонкий helper, вызывается из уже существующих узлов, без изменения их логики:**

```python
def emit_trace(node: str, event_type: str, content, *, run_id: str, iteration: int = 0, branch_id: str | None = None):
    event = {
        "run_id": run_id, "branch_id": branch_id, "iteration": iteration,
        "node": node, "type": event_type, "content": content, "ts": time.time()
    }
    trace_store.append(run_id, event)   # запись в персистентное хранилище — для отчёта после запуска
    trace_bus.publish(run_id, event)    # публикация в pub/sub — для живой доставки на фронтенд
```

Точки интеграции (по одной строке кода в каждый существующий узел, без переписывания их логики):
   - `planner.py`: `mode_reasoning`/`corpus_reasoning` → `type="thought"`;
   - `execution.py`: каждый элемент `planned_actions` → отдельный `type="action"`;
   - `orchestrator.py`: каждый ответ API (в идеале — уже сжатая версия из `evidence_compressor` Концепции 1) → `type="observation"`;
   - `evidence.py`: `aggregator_reasoning` → `type="thought"`;
   - `mode_router`/`evidence_router` в `graph.py`: исход роутинга → `type="decision"` (сейчас это уже есть как `print()`-строки, нужно просто продублировать в `emit_trace`).

**2.3. Доставка на фронтенд сайта — SSE, не через state.**
Эндпоинт `/research/{run_id}/trace` (например, на FastAPI), подписанный на `trace_bus`, пушит каждое новое событие в браузер в реальном времени по мере выполнения графа. Это необходимо, поскольку одна итерация может занимать 10–30+ секунд (несколько последовательных LLM-вызовов + внешние API-вызовы) — без живого стрима пользователь видит просто зависший спиннер. Альтернатива/дополнение: встроенный в LangGraph/LangChain `astream_events` даёт события уровня фреймворка (`on_chain_start`, `on_chat_model_stream`, `on_tool_start`, `on_tool_end` и т.д.) бесплатно, без ручной инструментации — но не даёт уже распарсенные, семантически типизированные reasoning-поля (`mode_reasoning` и т.п.), которые даёт ручной `emit_trace`. Для этого конкретного проекта ручной `emit_trace` предпочтительнее как основной механизм, поскольку ценный контент уже существует в виде чистых JSON-полей в коде, а не в виде сырого потока токенов.

**2.4. Паттерн UI** (адаптация thinking-режима под несколько отдельных LLM-вызовов за один прогон, а не единый chain-of-thought):
   - вертикальная лента свёрнутых по умолчанию секций, по одной на итерацию (например: «Итерация 2: донабор данных по SPOKEN»);
   - внутри секции — упорядоченные карточки: мысль (`reasoning` execution_planner) → действия (список `planned_actions` с параметрами) → наблюдения (сжатые результаты API) → мысль (`aggregator_reasoning`: цель достигнута? нужен реплан?) → решение (исход роутера: продолжить / реплан / финиш);
   - при активном fan-out из Концепции 1 — карточки группируются по `branch_id` (вкладки или параллельные колонки на корпус), сходящиеся в отдельную карточку «синтез сравнения»;
   - каждая карточка-наблюдение ссылается на `evidence_id` (из структурированного `Fact` Концепции 1) — обеспечивает drill-down к сырым данным прямо из ленты мыслей.

**2.5. Персистентный «отчёт по исследованию» — отдельно от живого стрима.**
После завершения графа (`assistant_node → END`) собрать всё в самостоятельный документ:

```python
class ResearchReport(TypedDict):
    run_id: str
    question: str
    final_answer: str
    facts: list[Fact]          # из Концепции 1
    trace: list[TraceEvent]     # из Концепции 2
    corpora_used: list[str]
    duration_sec: float
```

Это должно храниться и быть доступно долгосрочно (например, список «мои прошлые исследования» на сайте), независимо от того, жив ли чекпоинтер графа (`MemorySaver`, который по Концепции 1 должен стать `PostgresSaver`/`SqliteSaver`).

### Рекомендованный порядок внедрения
1. Добавить `emit_trace()` вызовы, захватывающие уже вычисляемые reasoning-поля (минимальный риск, ~час работы) — уже даёт видимую пользу без новых LLM-вызовов.
2. Построить SSE-эндпоинт + фронтенд-аккордеон/ленту, потребляющую живой поток трассы — это и есть собственно UX «наблюдать за мышлением агента».
3. Только после того как пункты 1–2 работают и проверены, рассмотреть переключение `planner`/`evidence_aggregator` на reasoning-модель (`deepseek-reasoner`) для настоящего token-level chain-of-thought — с учётом возросшей стоимости/задержки.

## Как две концепции стыкуются друг с другом

- `branch_id` в `TraceEvent` (Концепция 2) — это тот же идентификатор, что и ветки fan-out по под-целям `(lemma, corpus)` из Концепции 1.
- `evidence_id` в структурированном `Fact` (Концепция 1) — тот же идентификатор, на который должны ссылаться `observation`-события трассы (Концепция 2). Это даёт единый сквозной способ переходить от финальной таблицы фактов ИЛИ от живой ленты мыслей — к одним и тем же сырым данным.
- Обе концепции опираются на один и тот же общий принцип: **чётко разделять то, что должно попадать в контекст LLM** (компактное, структурированное, скоуп-отфильтрованное), **от того, что предназначено только для человека на выходе** (трасса, сырые артефакты, полный список фактов). Ни один сайд-канальный/выходной артефакт не должен незаметно повторно попадать в будущий промпт без явной фильтрации/ретрива.
